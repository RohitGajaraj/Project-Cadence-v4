SECTION 5: PARALLEL DEVELOPMENT MODEL (No code silos, document-driven handoff)
================================================================================

NO CODE OWNERSHIP. All tools are equal. Whoever is working owns that task.

Full detail: docs/decisions/parallel-development-model.md

PRINCIPLE:
You (Lovable) can work on ANY file in the codebase. So can Claude Code,
Antigravity, and Gemini. There are no restricted files. There are no "Claude
Code only" or "Lovable only" layers. This is a multi-tool workspace.

RESPONSIBILITY:
Whoever touches a file or task becomes its temporary owner. Your job is to:
  1. Document what you are doing (in the SSOT section 0 / live cursor, status board)
  2. Document WHY you are doing it (thought process, decisions)
  3. Document what you DID (what changed, why it changed)
  4. Document what is NEXT (what the incoming tool should do)
  5. Leave the codebase in a known state for the next tool

THE HANDOFF MECHANISM (replaces code ownership):
  - SSOT section 0 / live cursor (docs/planning/SOURCE-OF-TRUTH.md) — exact checklist of what is in-flight, what is done
  - Live status board (docs/planning/feature-dashboard.md top) — current task ID, blocks
  - docs/strategy/session-decisions.md — major strategic decisions made
  - Git commits — clear messages explaining WHY changes were made
  - Docs — architecture/frontend.md, design.md, AGENTS.md kept true with code

CONFLICT AVOIDANCE:
  - Always check the Live status board BEFORE you start
  - If another tool is actively working on the same file: STOP and wait
  - Commit often (5-15 min), push immediately so other tools see your work
  - When you end, leave the SSOT section 0 (the live cursor) true so the next tool knows where you stopped

FILES ARE SHARED. DOCUMENTATION IS THE BOUNDARY.

EVERY TOOL (Lovable, Claude Code, Antigravity, Gemini) OWNS:
  - SSOT section 0 / live cursor (docs/planning/SOURCE-OF-TRUTH.md) — keep it true
  - status board (docs/planning/feature-dashboard.md) — update it
  - code quality — write clean, documented code
  - code documentation — keep design.md, architecture/*, ENTRY.md in sync
  - handoff documentation — leave the next tool with full context

NO TOOL EXCLUSIVELY OWNS:
  - Any file in src/
  - Any file in supabase/
  - Any doc in architecture/
  - Any layer or "responsibility"

SECTION 9: TECH STACK & CONSTRAINTS (reference)
================================================================================

TECH STACK (do not change without explicit decision):
  - React 19 + TanStack Start (full-stack framework)
  - Vite 7 (build tool)
  - Tailwind v4 (styling)
  - shadcn/ui (component library, new-york variant)
  - Radix (headless UI primitives)
  - Lucide (icons — no mixed icon sets)
  - TanStack Query (data fetching)
  - Framer Motion / Motion (animations)
  - Supabase (database — read-only from Lovable's side)

TECH STACK DECISION RATIONALE:
The existing stack was chosen deliberately for co-development across Lovable,
Claude Code, Antigravity, and Gemini. Changing the stack risks breaking the
co-development workflow. Any tech change must preserve Lovable compatibility.
Full rationale: docs/decisions/tech-stack.md

HARD LIMITS — these break the co-development workflow for all tools:
  - Do not force-push to main (destroys committed work from other tools)
  - Do not use long-lived feature branches (all tools work on main)
  - Do not change the core tech stack without a decision doc in docs/decisions/

PATTERNS TO FOLLOW:
  - Use semantic Tailwind tokens from design.md (never hex literals)
  - Loader + Suspense for data loading (not useEffect + fetch)
  - TanStack Query for server state (useQuery, useMutation with queryKey)
  - TanStack Server Functions for API calls (import from src/lib/*.functions.ts)
  - Motion via Framer Motion (respect prefers-reduced-motion)
  - Error boundaries on every route

SECTION 13: PRODUCT AGENT ROSTER (what agents are running in Cadence)
================================================================================

Cadence runs 6 durable specialist agents (one per lifecycle band) plus ephemeral
specialists spawned by the Orchestrator for sub-tasks. Full roster: plan.md section 6.

DURABLE AGENTS:
  - Orchestrator: coordinates missions, spawns specialists, manages approval routing
  - Discovery: signal ingest, clustering, themes, opportunities, continuous feed
  - Strategist: cited PRD/spec, task graph, roadmap, anti-feature-factory guardrail
  - Builder: autonomous coding, test generation + QA gating, PR/deploy/release
  - Growth: launch/GTM/pricing assets, support triage (closes loop to Discovery)
  - Analyst: outcome measurement, re-score opportunities, insight memo, learn loop

EPHEMERAL SPECIALISTS: spawned by Orchestrator per task (competitor scan, UX
critique, stakeholder brief, data analysis, etc.). No permanent seats.

TRUST ARC (product behavior, not a design spec):
Agents start with closer governance (operator approves frequently) and earn
autonomy through demonstrated performance. Four stages: Observing -> Proving ->
Trusted -> Ambient. The Agent Trust Score and Autonomy Dial make this visible.
Full feature spec: docs/planning/feature-backlog.md.
