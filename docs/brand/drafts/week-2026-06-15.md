Green tests are not proof your code is right. They are proof your assumptions agree with each other.

This week two independent reviewers, whose only job was to disagree with my diff, caught a bug my passing tests missed. The tests fed clean literals. Production sends strings.

**LinkedIn**

A small debugging lesson from this week that I keep relearning.

We had a metric reading 0. The data was correct, the function looked correct, the tests were green. Everything agreed, and everything was wrong. A database value arrives as a string, not a number, and a type check quietly rejected every real row.

The unit tests missed it because they fed the clean shape, not the messy one production actually sends. Green tests prove your assumptions are consistent with each other. They do not prove the assumptions are true.

What caught it was an adversarial review: a second pass whose only job was to disagree. Tests check the code. A skeptic checks the thinking behind it. I am adding one to every meaningful change now.

**Title:** Honest instrumentation: three things building an agent product taught me this week

**Body**

Honest take after a week of building: the hardest part of an AI product is not the model. It is being honest about what your own system actually does.

Three moments this week made that concrete.

One. A metric on our dashboard read 0. The data was right, the code looked right, the tests were green. The cause was a serialization detail: a database number arrives over the wire as a string, and a type check silently rejected every real row. The tests passed because they fed clean numbers, not the messy shape the database actually sends. Green tests prove your assumptions agree with each other, not that they are true.

Two. We shipped a memory metric and asked the uncomfortable question. Not how much the system stores, but how much it reads back. A store you reopen compounds. A store you never reopen is a logfile in a trench coat. Reads, not writes, is the honest moat number.

Three. We left a number off the dashboard on purpose. Net revenue retention needs billing we have not built yet, so we left it blank with a one-line note instead of a plausible figure. A blank you can explain beats a number you cannot defend.

The thread through all three is one rule we hold the product to: the claim never outruns the wiring. If the interface says it, the code does it. If the chart shows it, the data backs it. It is not modesty. It is the cheapest way to build something people can actually rely on.
