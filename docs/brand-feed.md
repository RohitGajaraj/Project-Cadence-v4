## What these become, and the mix (context for whoever captures here)

High signal, low noise: a typical week of building yields only a few entries worth capturing. Do not dump routine work, status updates, or every change here. When in doubt, leave it out.

- It teaches a non-obvious lesson, or reveals a real mechanism, decision, tradeoff, or honest mistake — something a smart reader gains 30 to 60 seconds of genuine value from.

- It is fresh enough to be interesting and not already covered in a recent post (check the de-dup line at the bottom).

- What actually happened: three changes, in order. (1) I moved from one branch to five git worktrees running in parallel, each its own copy of the project on a different, non-overlapping slice of the roadmap (five builders in five rooms of the same house, never colliding). Not blind trust, a line I drew on purpose because I know which everyday actions are safe and which are not. (3) Resilience and decision-split: the loop runs on /loop, pauses when it hits a usage limit and resumes itself, and only escalates the founder-only calls to me. Concrete proof from one night: it ran about 27 cycles on its own (roughly 7 in the most important product areas, 3 on security, the rest across everything else) and left a clean dashboard by morning of what it built and where each piece stood. I also push usage hard enough that a Claude Max 20x plan runs dry in two to two and a half days, so while it resets I switch to other models (GLM or Kimi) and keep going. I now run most of it from my phone: Claude lets me take over a session remotely, and when a session needs a real decision it notifies my phone, I weigh in, and it carries on.
- Angle: the non-obvious lesson for anyone trying to run AI agents unattended is that autonomy is not "turn the safety off and walk away," it is rails plus a decision-split. The leverage is a bounded bypass (everyday actions run free, a hard denylist blocks the irreversible ones), isolation so parallel agents never collide, a real test gate so nothing broken ships, a self-resuming loop so a usage limit does not end the night, and a clear line between what the machine may decide and what only you should. The emotional truth that makes it land: the hard part was not the code, it was learning to trust a system you built well enough to stop hovering over it. Watching something you built run without you is a strange kind of quiet.

- Angle: the non-obvious lesson for anyone building agents: every commodity capability your agent depends on is a potential single point of failure, and the move is not "pick the best vendor," it is "always have a native default that needs nobody." Build the floor first, wrap the paid provider behind a seam, and make the degraded path honest (don't fake the capability the cheaper backend lacks). An autonomous system you can't run without a specific vendor's key is not really autonomous. This is the same instinct as owning your data store instead of renting it: own the floor, rent the upside.

## 2026-06-21 - the honest way to prove your AI is getting smarter at one user, and why most "lift" numbers are a lie
- Pillar: AI-craft / honest-metrics
- Context (the full story, so the engine has everything): my product's whole bet is that it accumulates decision memory and gets better at a user's judgment over time. That claim needs proof, and it needs to hold at ONE user (an investor will not wait for a cohort). The tempting move is a "memory lift: +X%" number on a dashboard. But here is the trap almost everyone falls into: a real causal lift figure requires an A/B control, the same decision made WITH memory and WITHOUT, and nobody has that. So the number you see on most "our AI learns" dashboards is either invented, or it is a before/after over calendar time dressed up as if the AI caused it, when the person simply got better with practice. I had already shipped a card that showed the raw "share of your bets that validated" and deliberately REFUSED to call it lift, exactly because I did not have a control. The task was to build the lift half without lying.
- What actually happened: I built it as an observational split instead of a fake experiment. For each past decision the user reviewed, I measured how much memory had accumulated when that decision was made, then split their decisions into the half made with shallow memory vs the half made with deep memory and compared how often each half validated. That is real evidence of association from data you already have, no experiment needed. The hard part was the honesty gates, not the math: it refuses to print a number unless there are enough reviewed decisions on each side (so one outcome flipping cannot swing it), unless the two halves actually differ in memory depth (or there is nothing to compare), AND unless the difference clears a real statistical significance test (a 95% confidence interval, computed with no library) so it never reports noise as a finding. Below any gate it says "not enough data yet" with the specific reason, never a fabricated figure. And because "decided later with more memory" is tangled up with "decided later, when I was also just more experienced", the card NAMES that confound right next to the number and stays color-neutral whether the lift is positive or negative, so it reads as an association to interpret, not a victory lap. A negative result is shown exactly as-is.
- Angle: the non-obvious lesson for anyone shipping an "our AI learns / personalizes / compounds" claim: you almost never have the controlled experiment your headline implies, and faking it is the fastest way to lose a technical user's trust. The honest alternative is an OBSERVATIONAL cut of data you already have, gated hard against small-sample noise (a real significance test, not vibes), with the confounds named at the pixel and a neutral frame. "Correlational, here is the caveat" beats "+37% lift" the moment someone smart asks how you measured it. Trustworthy beats impressive, and at one user trustworthy is the only thing you actually have.
- Facts to keep exact: the metric splits a user's reviewed decisions into a lower-memory-depth half and a higher-memory-depth half and compares validated-outcome share; it gates on a minimum sample per half (so a single outcome cannot swing it), a minimum difference in memory depth between the halves, and a two-proportion 95% confidence interval (no external stats library), and otherwise renders "not enough data yet" with the reason; it reports whole percentage points (a decimal at that sample size is false precision), shows negatives as-is, stays color-neutral either direction, and names the time/practice/survivorship confound inline. The whole design was pressure-tested by adversarial reviewer agents whose entire job was to find where it overclaimed. Do not name the product or the surface.

## 2026-06-21 - I shipped a "delete everything about this customer" button turned OFF, and the review taught me the off-switch protects nothing if the bug is in the delete itself
- Pillar: build-detail
- Context (the full story, so the engine has everything): I built the legally-required "right to be forgotten" eraser, the thing that wipes every trace of a customer's data on request. It is the most dangerous code in the whole product: one cascade across dozens of tables, irreversible, and if it is even slightly wrong it either fails to delete (a legal problem) or deletes too much (a catastrophe). The safe way to ship something this dangerous is to ship it dormant: behind an off-by-default flag, runnable only by a privileged operator, with the real destructive run staying a deliberate human act. I did all of that. I felt safe. Then I ran it past four independent adversarial reviewers, each told to try to break it from a different angle, and the most useful finding flipped my mental model of what "safe" even meant here.
- What actually happened: two design choices are worth the post on their own. First, the naive way to erase a tenant is "delete the parent row and let the database cascade." I checked the live schema and that is quietly wrong: of the ~60 tables holding tenant data, a third would BLOCK the delete (the database refuses), and a handful would just NULL out the link instead of deleting, leaving the most valuable data (the accumulated machine memory) sitting there orphaned but present. The worst outcome of a "forget me" feature is data that looks deleted but isn't. So instead of trusting cascades, I made the eraser discover every table that carries a tenant id straight from the database's own catalog and delete from each explicitly, in a loop that resolves the dependency order itself and refuses to finish silently if anything is left. The nice side effect: it can never go stale, because it reads the live list of tables every run, so a table someone adds next year is covered automatically. Second, and this is the lesson: I had written in the file's header that the eraser is "single-tenant by construction, it can only ever touch the target customer." The reviewer proved that claim was a lie. The function took the column to filter by as a parameter, and nothing checked it. A typo, or a future caller passing the wrong column, would turn "delete this one customer" into "delete by something else entirely," and because this code runs with god-mode database privileges that bypass the normal per-customer security, there was no backstop. The dormant flag did not help. The operator-only access did not help. Both guard the front door; the bug was in the room. The fix was three lines: a hard allow-list of exactly which columns are valid to delete by, plus making the dangerous inner function check the off-switch too, not just the public wrappers. I also folded a crash in the very verification step an operator would run to confirm the deletion worked.
- Angle: the non-obvious lesson for anyone shipping dangerous, irreversible operations: "we ship it disabled and locked behind admin access" makes you feel safe, but those are perimeter guards. They do nothing about a bug inside the destructive logic itself, which is exactly where the catastrophe lives. A dormant feature with a flaw in its core is still a loaded gun, the safety just delays the trigger. Put the real invariant where the damage happens (here: hard-enforce the single-tenant filter inside the function, do not just claim it in a comment), and have someone actively try to disprove your safety claims, because the most dangerous bug is the one your own header file confidently says cannot exist.
- Facts to keep exact: it is the GDPR/CCPA right-to-be-forgotten erase cascade, shipped DORMANT (off by default flag) and runnable only by a privileged service role; verified against the live schema that of ~60 tenant-data tables roughly a third have delete-blocking constraints and 6 would null-out rather than delete (orphaning the agent memory); the eraser reads the table list dynamically from the database catalog so it cannot go stale; the most serious review finding was an uncontrolled filter-column parameter that could have widened the delete beyond one tenant under privilege-bypassing access, fixed with a column allow-list + an inner dormancy check; a four-lens adversarial review folded four must-fix issues total; the engine was dry-run-verified on the real production schema with zero destruction (deleted nothing real, using a non-existent id, inside a rolled-back transaction); it deliberately does TENANT-level erasure now and defers true PERSON-level identity erasure as a separate, legally-scoped follow-up. ## 2026-06-20 - the "memory moat" everyone talks about is not a screen you visit; it is your own past decisions showing up uninvited at the moment you decide again
- Pillar: strategy/taste
- Context (the full story, so the engine has everything): the thing my product is supposed to be defensible on is that it remembers the outcomes of past decisions, so it gets sharper the longer you use it. For months that "memory" lived where memory features usually live: a section you could open and search. Useful, but it sits there waiting for you to remember to ask it. This week I shipped the first increment of a different idea, and writing it changed how I think about what a moat actually is. The insight is that memory is only a moat when it is ambient: it has to surface itself, unprompted, exactly where a decision is being made, before the user thinks to look. So instead of a smarter memory page, I built a small, calm, dismissible prompt that appears right next to wherever you are weighing a call, and quietly says "you reasoned this way before, and here is how it actually turned out," with the relevant past outcomes pulled by meaning, not keyword.
- What actually happened: the whole increment reused data I already had (the outcomes I was already storing, already embedded), so there was no new database schema and no new capture step for the user. The engine embeds the text of whatever decision you are looking at, finds the most semantically similar past outcomes, keeps only the genuinely relevant ones above a similarity bar, caps it to a few, and renders them in place. It is fail-safe by construction: if there is no real precedent, it shows nothing, so it never becomes noise. I wired it at the three places a call is most explicitly made (the bet, the spec, the red-team review). The hardest design decision was not technical, it was restraint: where NOT to show it (a list of many items would have spammed one prompt per row and made the product feel busy), and how to render nothing gracefully so the surface stays calm.
