# Build-in-public insight feed (one way → the build-in-public repo)

> _Created: 2026-06-15 · Last updated: 2026-06-18_

This is the single capture point for postable insights from building Cadence. As we build (any tool, any session), real, non-obvious, public-safe moments get appended here: a decision, a mechanism, a real number, a lesson, a launch reaction, a screenshot worth grabbing. The private **build-in-public** repo's weekly engine and real-time watch read this file first, before scouting, so the posts are grounded in what actually happened.

## What these become, and the mix (context for whoever captures here)

**Only solid, genuinely postable insights belong here. This is a high bar, not a build log.** If it would not make a genuinely good, interesting social post on its own, it does not go in. High signal, low noise: a typical week of building yields only a few entries worth capturing. Do not dump routine work, status updates, or every change here. When in doubt, leave it out.

**What qualifies (the bar). An entry should clear most of these:**
- It teaches a non-obvious lesson, or reveals a real mechanism, decision, tradeoff, or honest mistake — something a smart reader gains 30 to 60 seconds of genuine value from.
- It is grounded in something that actually happened, with exact facts (real numbers, real behavior). Never invent or round for effect.
- It is public-safe: no secrets, credentials, customer data, or anything that breaks product stealth or current-job safety.
- It is fresh enough to be interesting and not already covered in a recent post (check the de-dup line at the bottom).
- It stands on its own as a post, without naming the product or the audience.

If it clears the bar, **capture it comprehensively**: full context and exact facts, written close to ready-to-post so the engine (and the other repo) has the whole story and can draft without re-deriving anything. Do not compress a good insight into one terse line; give it the context, the method, the result, and the lesson. **And attach the real proof, not just a description of it:** when a screenshot or short screen recording strengthens the post, SAVE the actual file under `docs/screenshots/brand-feed/` (local-only, gitignored) and reference its path in the entry's capture cue. Still note any extra cue the founder should grab himself in Buffer (a link, an account to tag), and flag anything that must be redacted for stealth before posting. Note "none" only if no proof helps at all.

These entries become posts in the founder's private **build-in-public** system, so capture them with that in mind. The voice there is **lived first person (never "we"), hook-first, no call to action, stealth on the product, grounded in real fact, and humanized**. (The full binding rules, `voice-profile.md` and `knowledge.md`, live in the SEPARATE private `build-in-public` repo, not in this one. The engine runs with both repos checked out, so at run time it reads those rules together with this feed; the brief cue in this sentence is all you need when capturing here.) You do not write the final, voiced post here (the engine does that, in the founder's voice, reading both repos). But capture the real insight, its full context, its angle, and the exact facts comprehensively and accurately, close to ready-to-post, so the engine has everything it needs and nothing has to be reconstructed.

**Not every post comes from this feed, and that is by design.** The week is a deliberate **mix**: roughly half from real build work (this feed) and half from the wider world (market moves, model and tool launches, product-management shifts, early-adopter takes). So this feed is one input, not the only one. Capture what is genuinely notable and postable; never force a small or internal detail into a post just to fill the feed.

**Audit trail:** entries are dated, newest first, and carry a `Status` (`ready` until posted, then `used`), so this file doubles as the record of what has been drawn from and what is still available.

- What actually happened: First, the database migration had an index with a "where this row has not expired yet" condition that used the current time. The type checker has no idea; it only fails when the migration actually runs. Second, the cache never actually cached: the code checked for a cached answer, but then called the expensive service anyway, every time, because the "skip the call if we already have it" branch was wired around the wrong block. It saved nothing. Third, the lookup compared the expiry against the literal text "now()" instead of an actual timestamp, so every read silently errored and fell through to a miss. I fixed all three, re-ran the migration against a throwaway production copy to confirm it now applies cleanly, and shipped the fix instead of a duplicate.
- Angle: "the build is green" is not "it works", and the gap is widest exactly where it is most dangerous. A type checker cannot tell you a database migration will be rejected at deploy time, or that a cache silently does nothing, or that a query always errors and quietly falls back. The only things that caught all three were running the migration against a real, disposable copy of the database before trusting it, and reading the control flow like an adversary instead of like its author. Speed without that floor just means you ship broken things faster, and more confidently.

## 2026-06-18 - I added a stop button to my autonomous agents, and learned that "stop" is never one switch
- Pillar: build-detail
- Context (the full story, so the engine has everything): I build a system where AI agents run multi-step "missions" mostly on their own: a background loop keeps advancing each mission a step at a time, and a separate scheduled job resumes any work that paused. I wanted a real brake, a "cancel this mission" button, so a human can stop one mid-flight. The obvious fear was that bolting a stop onto a self-driving loop would mean surgery on the loop itself, which is exactly the kind of change that quietly breaks autonomy. It turned out the opposite was true, and the genuinely interesting lesson was hiding one layer down.
- What actually happened: First the good surprise. Before writing anything I traced how the loop actually decides what to work on. The advance step only ever picks up missions whose status is "running," and it bails immediately on anything that is not. So "cancel" did not need a single new line in the loop, just flip the mission's status and the loop ignores it forever. Zero changes to the dangerous part. But then the real catch. A second job resumes individual sub-runs by looking at THEIR status, not the mission's, so cancelling the parent alone would have left orphan children that the resumer happily woke back up. So the cancel has to reach down and stop the children too. And the part I am proud of catching: there is a database trigger that releases a build's file locks (so two agents never edit the same file at once) when a run reaches a terminal state, but it only fired on "completed," "halted," and "failed," never on "cancelled." Cancelling a build mission would have silently left its file locks held forever, quietly blocking every future build on those files, with no error anywhere. I only found it by reading the trigger before trusting it. So the cancel explicitly releases those locks too. I also added a guard so that if a mission happens to finish in the split second between the click and the write, its honest "completed" is not overwritten with "cancelled."
- Angle: in an autonomous system, "stop" is never a single switch. The loop, the resumer, the lock-keeper, and the metrics each have their own idea of what "done" means, and a real stop has to satisfy all of them at once. The headline ("I added a cancel button") took ten minutes; the value was the hour of reading every subsystem that reacts to a terminal state and making them agree. Distributed stop is a consensus problem dressed up as a button.
- Facts to keep exact: the advance loop only progresses missions in status "running"/"in_progress" and no-ops on the rest, so cancelling needed zero loop changes; a separate resume job selects sub-runs by their own status ("queued"/"running"/"waiting_approval"), so the parent's children must be cancelled explicitly; a DB trigger released file locks only on "completed"/"halted"/"failed", never "cancelled", so cancel had to release them by hand or they orphan and block future builds; a race-guard prevents overwriting a just-finished mission's terminal state; no schema migration was needed; the whole change was three files plus its docs and passed typecheck + build + an adversarial self-review.
