## Part A. Current architecture (verified, do not duplicate the contracts)

This part is a navigational map. Where a contract already exists in `architecture/`, this section says what is true and points there rather than restating it.

### A.1 Stack and request path

TanStack Start (full-stack React 19) on Vite 7, deployed to a Cloudflare Worker. The SSR entry is rerouted to `src/server.ts`, which wraps `@tanstack/react-start/server-entry` and catches h3-swallowed 500s (the `{unhandled:true, message:"HTTPError"}` body pattern) via `normalizeCatastrophicSsrResponse`, rendering a branded error page instead. Treat `src/server.ts` as load-bearing.

Routing is file-based in `src/routes/`. `_authenticated.*` is the gated app shell, one route per surface. `p.$slug.tsx` is the public prototype share. `src/routes/api/*` are server routes that use `createFileRoute` with `server.handlers.{POST|OPTIONS}` (no React component). `src/routes/routeTree.gen.ts` is generated; never hand-edit.

Server logic lives in `src/lib/*.functions.ts`, one TanStack server-function module per domain. Files suffixed `.server.ts` run in the Worker only and never reach the client bundle; importing one from a client component fails at build. Full frontend contract: [`../../architecture/frontend.md`](../../architecture/frontend.md).

### A.2 Data model and RLS

Supabase Postgres with RLS migrations in `supabase/migrations/` (timestamped SQL, RLS-aware, applied in timestamp order). `workspaces` is the multi-tenancy anchor; every table with a `workspace_id` FK is RLS-scoped to workspace membership. The agent and orchestration spine: `agents`, `agent_runs`, `agent_run_checkpoints`, `missions`, `mission_steps`, `agent_messages`, `agent_approvals`, `agent_tools`, `agent_autonomy`, `agent_memory`. Telemetry and governance: `ai_events`, `ai_budgets`, `ai_surface_budgets`, `ai_budget_alerts`, `ai_evals`, `guardrail_rules`, `guardrail_hits`, `tool_calls`. Ingest and reactor: `signals`, `event_queue`, `event_subscriptions`, `ingest_tokens`. Product artifacts: `prds`, `opportunities`, `themes`, `decisions`, `learnings`, `prototypes`/`prototype_files`, `drift_snapshots`, `artifact_lineage`. Connectors: `connections`, `connection_secrets`, `connection_bindings`. Build engine: `studio_changesets`, `studio_changes`, `builder_file_claims`. Retrieval and voice: `rag_chunks`, `workspace_briefs`, `profiles.voice_anchor_text`. Governance gate: `kill_switch` (read via `current_kill_state` RPC) and `user_api_keys` (BYO keys).

Two security invariants are load-bearing and graded in Part B context. The only two anon-readable surfaces, `decisions` and `prototypes`/`prototype_files`, are hardened with column-scoped GRANTs, RLS policies scoped `TO anon USING (is_public = true)`, and removal from the `supabase_realtime` publication (KI-17, migration `20260614180000`). The signup path is RLS-and-trigger-sensitive and is the subject of KI-13. Full data and security contracts: [`../../architecture/data.md`](../../architecture/data.md), [`../../architecture/security.md`](../../architecture/security.md).

### A.3 The AI runtime chokepoint

Every model call goes through `src/lib/ai/runtime.server.ts`. There is no second path. The two exported variants are `callModel` (awaited JSON, used by the agent loop) and `callModelStream` (SSE, used by `chat.ts` and `studio-chat.ts`). Each runs the same ordered pipeline: governance halt check (`checkKillSwitch` then `checkMissionCaps`, both before any spend, throwing a typed `GovernanceHaltError`), budget and per-surface budget checks, optional prompt resolution, optional RAG retrieval, input guardrails, three-tier provider routing (explicit `byoOverride`, then a matching key from `user_api_keys`, then the Lovable AI gateway, with a local-dev `google/*` fallback through `GEMINI_API_KEY`), output guardrails, cost accounting (`estimateCostUsd` then `incrementBudget`, `incrementSurfaceBudget`, `record_mission_usage`), and persistence into `ai_events` (always) plus `guardrail_hits`.

The `CallSurface` union is the closed list of allowed surfaces: `"agent" | "chat" | "copilot" | "prd" | "discovery" | "studio" | "brief" | "eval" | "judge" | "embed" | "scheduler" | "test"`. Adding a new AI surface means adding a literal here, not calling the gateway directly. Note for M-D: the public-API and MCP work needs new surface literals (`mcp_server`, `a2a`) that the runtime contract already anticipates but the union does not yet carry. Full chokepoint contract: [`../../architecture/runtime.md`](../../architecture/runtime.md).

### A.4 The agent loop and orchestration

`src/lib/ai/loop.server.ts` runs the planning loop. `runAgentLoop` resolves the agent by slug, applies workspace backpressure (queue past 5 running runs), creates an `agent_runs` row, loads enabled `agent_tools` filtered to `TOOL_REGISTRY`, recalls memory (`recallMemoryRefs`), loads the brief and voice anchor, consumes any inbound handoff, builds the system prompt (with tool output wrapped in `<untrusted_tool_output>` tags and a prompt-injection guard), loads the agent's arc, and calls `executeLoop`. `executeLoop` computes `maxSteps` via `adaptiveStepBudget`, checkpoints each step into `agent_run_checkpoints`, calls `callModel` with `responseFormat: 'json_object'`, parses `{thought, action}`, applies the safety floors (`HIGH_RISK_FORCE_REVIEW`, `HIGH_RISK_MIN_CONFIRM`, `PAUSE_ON_APPROVAL_TOOLS`), composes the per-tool mode with the arc via `resolveApprovalMode`, and either gates the call into `agent_approvals` or runs it through `withIdempotency`. `resumeAgentLoop` and `executeApproval` carry the durable-resume and post-approval paths.

Mission advance is deterministic and model-free. `advanceMissionCore` in `mission-advance.server.ts` runs every minute from the `resume-runs` cron: it reflects step status from child run status, requeues failed steps with backoff, dispatches dependency-ready steps via the `next_ready_mission_steps` RPC under compare-and-swap, threads recalled `memory_refs` into each handoff, and finalizes. Handoff is the typed A2A path: `handoff.server.ts` enqueues `agent_messages` with a `HandoffPayload` (task, context, artifacts, open_questions, constraints, memory_refs), validates memory_refs against real `agent_memory` ids (phantom-stripped), and `maybeCompleteMission` closes the mission and auto-captures a `decisions` row when no unconsumed messages remain and all steps are terminal. Memory recall, outcome distillation (`rememberOutcome`), adaptive budgets (`budget.ts`), bounded retry (`retry.ts`), and the trust arc (`trust.server.ts`: `resolveApprovalMode`, `computeAllAgentTrust`, `suggestArc`) are all live. Full orchestration contract: [`../../architecture/orchestration.md`](../../architecture/orchestration.md).

### A.5 The tool registry and approval modes

Tools are declared in `src/lib/ai/tools/registry.server.ts` with Zod-validated args. Categories: read tools (`workspace.search`, `repo.read`, `web.search`, and so on), write tools (`tasks.create`, `calendar.create`, `github.pr.open`, the `studio.*` family), memory tools (`memory.remember`, `memory.reflect`, `memory.promote`), and planning tools (`mission.plan`, `mission.dispatch`, `agent.handoff`, and the rest). Per-user approval mode (`auto` / `confirm` / `review`) lives in `agent_tools.mode` and is composed with the agent's arc at execution. Safety floors are hard-coded and never loosened: `studio.pr.merge` is always `review`; `calendar.create`, `studio.commit`, `studio.pr.open` are at least `confirm`; the `studio.*` write trio pauses the run for approval. The `mission.plan` tool validates every planned `agent_slug` against the live roster before inserting `mission_steps` (this is the validation the orchestrator slug bug trips; see B.2).

### A.6 API routes and cron hooks

Streaming chat is `src/routes/api/chat.ts` (surface `chat`, `callModelStream`). Public ingest is `api/public/ingest-signals.ts` (bearer-token auth against `ingest_tokens`, inserts into `signals` with explicit `workspace_id` so the reactor trigger fans out). The A2A agent card is `api/public/a2a.agents.cadence.card.ts` (unauthenticated GET). OAuth callback is `api/public/connect/github/callback.ts`. Cron hooks under `api/public/hooks/` authenticate via `requireHookCaller` (matches `apikey`/`x-cron-key`/Bearer against `SUPABASE_PUBLISHABLE_KEY`) and use the service-role `supabaseAdmin` client: `resume-runs` (every minute; promotes runs, then advances up to 20 missions), `approvals-tick`, `event-reactor-tick`, `outcome-tick` (hourly), `indexer-tick` (hourly), `eval-tick`, `eval-suite-tick` (daily 3am), `drift-tick` (daily 4am), `memory-tick` (daily decay sweep).

### A.7 The connector platform

`src/lib/connectors/` holds a typed provider registry (`registry.ts`), the credential chain (`resolve.server.ts`), the secret vault (`crypto.server.ts`, AES-256-GCM, service-role only), and provider adapters (`providers/github.server.ts`). Founder policy: user-facing connectors are OAuth-only, no key paste. `CONNECTOR_REGISTRY` lists 9 providers; GitHub uses GitHub App auth, the rest use the Lovable `oauth_gateway`, and `firecrawl` is platform-internal env-only. `resolveProviderAuth` is a three-tier chain that degrades on failure: workspace binding (`connection_bindings`), then the caller's own `connections` row, then an env-var fallback. Full integrations contract: [`../../architecture/integrations.md`](../../architecture/integrations.md).

### A.8 The env and secret split

Client (Vite prefix, browser-safe): `VITE_SUPABASE_URL`, `VITE_SUPABASE_PUBLISHABLE_KEY`. Server (wrangler secrets, never in the client bundle): `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY`, `SUPABASE_SERVICE_ROLE_KEY`, `LOVABLE_API_KEY`, `GEMINI_API_KEY` (local dev only), the GitHub App secrets (`GITHUB_APP_ID/CLIENT_ID/CLIENT_SECRET/PRIVATE_KEY/SLUG`) plus the legacy `GITHUB_TOKEN`/`GITHUB_REPO`, `FIRECRAWL_API_KEY`, and the per-connector OAuth client-ID env vars (`LINEAR_*`, `GOOGLE_*`, `NOTION_*`, `FIGMA_*`, `ATLASSIAN_*`, `MICROSOFT_*`). The split rule: never put a `VITE_` prefix on a secret. The connector OAuth client-ID secrets being unset is the operational blocker behind B.4.

### B.1 · `humanizeText()` runtime sanitizer · M-B · status: Missing/Planned

**What.** A pure, unit-tested `humanizeText(input: string): string` helper that normalizes em and en dashes to honest punctuation, strips every banned invisible Unicode character (zero-width space `U+200B`, ZWNJ `U+200C`, ZWJ `U+200D`, word joiner `U+2060`, BOM `U+FEFF`, non-breaking space `U+00A0`, narrow no-break `U+202F`, the exotic-space range `U+2002` to `U+200A`, soft hyphen `U+00AD`, directional marks `U+200E`/`U+200F`, replacement char `U+FFFD`), collapses double spaces, and trims trailing whitespace. It must skip fenced code blocks and inline code spans so a legitimate dash inside a code sample survives. It wires into the chokepoint on both paths: `callModel` sanitizes the final text before return; `callModelStream` sanitizes on a buffered token boundary so a multi-byte or code-fence sequence is never split mid-stream.

**Why.** The platform's promise is that its generated output (PRDs, drafts, chat, decision rationales) reads as if a sharp human wrote it. The system-prompt humanization directive in `prompts.server.ts` is convenience and models drift off it. The sanitizer is the hard gate: the boundary every word passes through, treated the way the chokepoint already treats guardrails and cost. This is a standing founder ruling (canon §13.8, [`../conventions/humanized-output.md`](../conventions/humanized-output.md) Level 2).

**Where in code.** New pure helper at `src/lib/ai/humanize.ts` (no I/O, no imports beyond standard string ops, so it is trivially unit-testable). Unit tests at `src/lib/ai/humanize.test.ts` covering: each invisible character class stripped; em and en dash replacement; a fenced block left untouched; an inline-code span left untouched; a dash adjacent to a code fence boundary handled correctly; idempotence (`humanizeText(humanizeText(x)) === humanizeText(x)`). Wiring in `src/lib/ai/runtime.server.ts`: call on the resolved text in `callModel` after output guardrails and before the return shape is built; in `callModelStream`, buffer on a safe boundary (newline or end-of-fence) and sanitize each flushed segment, with a final flush at stream close. The streamed path must preserve the existing post-stream `logAiEvent` and SSE meta-event contract.

**Acceptance criteria.** (1) `bun test src/lib/ai/humanize.test.ts` passes, including the idempotence and code-skip cases. (2) A `callModel` response containing an em dash and a zero-width space returns text with neither, while a fenced code block in the same response keeps its original characters byte-for-byte. (3) A streamed response renders identically clean to the user, with no token split across the sanitizer boundary (verified by a test that feeds a dash straddling two chunks). (4) The build-time grep gate (`rg "—|–"` and the invisible-character `rg -nP` from the convention) returns zero on a generated sample. (5) No measurable change to TTFT beyond the buffer boundary (the buffer is bounded to a line, not the whole response).

### B.2 · Orchestrator slug-bug fix · M-0 · status: Missing/Planned (fix first, pre-everything)

**What.** Align the orchestrator's planning prompt with the seeded roster so `mission.plan` stops rejecting phantom slugs. The live prompt (seed migrations `20260606121018_*.sql:33` and `20260606120924_*.sql:108`) instructs the planner: "Common slugs: discovery, strategist, builder, growth, analyst." The actually-seeded routable slugs are `discovery-scout`, `strategist`, `prd-writer`, `builder`, plus `orchestrator`. So `discovery`, `growth`, and `analyst` are phantoms. `mission.plan` validates every planned `agent_slug` against `new Set(roster.map(r => r.slug))` and rejects unknowns, so any multi-agent mission that includes a sensing step (the planner's natural first move) throws at plan time and the mission dies.

**Why.** This is a live mission-killer and it gates the entire "real loop, real data" thesis. Nothing downstream (ambient on-ramp, moat visibility, monetization) can be demonstrated on a real account while the first orchestrated mission crashes. Canon §2 and §7 both rank this fix as pre-everything.

**Where in code.** Two coordinated edits. (1) Correct the orchestrator prompt text in the seed migration path so the named slugs match the seeded roster (`discovery-scout`, `strategist`, `prd-writer`, `builder`) and the prompt instructs the planner to read the live roster rather than a hard-coded list. Because seed prompts are written at user-provisioning time, ship this as a new forward migration that updates existing `agents.system_prompt` rows for `slug='orchestrator'` and corrects the seed helper, rather than editing an already-applied migration. (2) Defense in depth in `registry.server.ts`: where `mission.plan` validates slugs, add a tolerant alias map (`discovery` -> `discovery-scout`, and reject-with-a-clear-error for genuinely absent roles like `growth`/`analyst` so the planner re-plans against the real roster instead of silently failing). The roster passed to the planner prompt should be generated from the live `agents` rows for the user, not a literal.

**Acceptance criteria.** (1) A multi-agent mission whose first step is a sensing/discovery hop plans and dispatches without `mission.plan` throwing. (2) The orchestrator prompt, read from a freshly provisioned account and from an existing account after the migration, names only seeded slugs. (3) A planner that emits a genuinely unknown slug gets a structured re-plan signal, not an uncaught throw. (4) The M-0 exit condition holds: a real new account runs a multi-agent mission without crashing (paired with B.3 and B.4).

### B.3 · Migration-sync and KI-13 owned apply-and-verify · M-0 · status: Partial

**What.** Get the six 2026-06-14 migrations applied and verified on the live database through an owned apply/verify step, not a passive wait on Lovable's sync. The set: KI-13 signup fix (`20260614140000`), the memory-recall scope COALESCE fix (`20260614091000`), the retry columns, ritual-sessions (`20260614150000`), the eval-score scale standardization (KI-14, `20260614160000`), and the decision-share surface (`20260614170000`, plus the KI-17 anon hardening `20260614180000`). KI-13 leads: live `POST /auth/v1/signup` returns 500 `Database error saving new user` because the `handle_new_user` trigger chain throws on the live DB (migration drift; a seed helper or table is absent), so no real account can be created platform-wide. The fix is landed defensively in code (each seed step now runs in its own `BEGIN..EXCEPTION` subtransaction so a failed seed logs a WARNING and signup still completes), but it is unverified live until the migration applies.

**Why.** The whole "real data" thesis is blocked at the door. Until signup works, there is no real account to run the loop, compound memory, or measure the gauntlet on. Until the COALESCE memory fix lands, the autonomous path recalls only reflections, not semantic memory, so the moat under-delivers on exactly the path that is supposed to prove it. Canon §2.2 and §12 M-0 name this an owned step with a named owner and a one-week manual-apply fallback.

**Where in code.** No new application code; this is an apply-and-verify operation against `supabase/migrations/`. The owned step: confirm the six migrations are present and ordered, apply them to live (manually within a week if the Lovable sync lags), then run the verification probes. Verification touchpoints: a fresh signup completes and self-heals profile plus workspace (KI-13); the memory recall path returns semantic `agent_memory` rows, not only reflections (COALESCE fix); eval scores read on the 0 to 100 scale across `eval_runs`, `eval_case_results`, and `drift_snapshots.avg_eval_score` (KI-14); a decision can be made public and read anon through the column-scoped grant only (KI-17). Track status in [`known-issues.md`](./known-issues.md), flipping each KI row in the same commit that verifies it.

**Acceptance criteria.** (1) A brand-new email completes signup end to end with a usable profile and workspace, verified on the live host. (2) An autonomous mission hop recalls at least one semantic memory (not just a reflection) into its handoff `memory_refs`. (3) The eval and drift surfaces render 0 to 100 scores with no overflow artifact. (4) KI-13, KI-14, KI-17, and the memory-scope row are flipped to resolved in `known-issues.md` with the verifying commit referenced. (5) An owner is named and a manual-apply fallback date is recorded if the sync has not landed.

### B.4 · Connector activation (OAuth registration + one live ingest source) · M-0 / M-A · status: Partial

**What.** Make at least one connector operational end to end and land one real ingest source. The connector platform is fully wired in code (registry, credential chain, encrypted vault, GitHub App adapter, the `oauth_gateway` path for the other eight providers), but no OAuth client IDs are registered, so SENSE is webhook-only in practice (KI-01, KI-12). The M-0 minimum is one real ingest source (the webhook door working, or one connector live). The M-A target is at least two real ingest sources.

**Why.** SENSE is the first station of the loop. Without a real input the loop cannot close on a real user's data, which is the M-A exit and the precondition for the moat to compound. Canon §2.3 and §12 name connectors as a specific, fixable gate.

**Where in code.** No core rewrite; this is registration plus verification against existing wiring. (1) Webhook path (fastest M-0 source): confirm the `F-V5-INGEST-WEBHOOK` migration (`20260611190000`) is applied (gated on the same sync as B.3), then verify `api/public/ingest-signals.ts` accepts a token, inserts into `signals` with `workspace_id`, and the `signals_reactor_fanout` trigger fans into `event_queue`. Note the open KI-10 risk: the endpoint has no rate limiting; add a per-token cap before any public exposure. (2) OAuth path (one live connector): founder registers the OAuth client for one provider and sets its `*_APP_USER_CONNECTOR_CLIENT_ID` wrangler secret (and for GitHub, the full App secret set plus the PKCS#8 key conversion per KI-12), then verify `resolveProviderAuth` resolves a real account through the workspace-binding or user-connection tier and one ingest pulls real records into `signals` or the relevant artifact table.

**Acceptance criteria.** (1) A POST to the ingest webhook with a valid token lands a `signals` row and produces an `event_queue` entry. (2) At least one OAuth connector completes the connect flow and `resolveProviderAuth` returns live credentials through a non-env tier. (3) One ingest run writes real external records into the workspace (M-0: one source; M-A: two). (4) A per-token rate cap exists on the public ingest endpoint before it is exposed (KI-10). (5) The connector card UI reflects the connected state.

### B.5 · Ambient-default on-ramp · M-A · status: Partial

**What.** Turn the felt experience from "review everything" into "ambient watch, approve by exception," through defaults and UX rather than new architecture. Today, real new users default to `observing` (every tool gated to `review`), which is the opposite of ambient; only demo accounts seed `trusted`. The on-ramp makes the observing -> proving -> trusted progression visible and earned, so gating loosens as an agent demonstrates safety. Concretely (canon §7): a Today queue of two card types, "Waiting for your call" (approve/override) and "Executed and learned" (summary plus one-click undo), where the green executed-and-learned share grows as decisions roll forward without override, and each override asks one question ("what's your reasoning?") that is stored as a memory signal.

**Why.** The product claims to be ambient and governed. With `observing` defaults it is governed but not ambient, so the core promise is unfelt for every real new user. This is mostly defaults plus UX, which is why it is M-A and not a deep rewrite. Canon §3 and §7.

**Where in code.** The arc machinery is Built and reused, not replaced: `agent_autonomy.arc`, `resolveApprovalMode` and `maybeAutoAdvanceArc` in `trust.server.ts`, and `computeAllAgentTrust`/`suggestArc`. The work: (1) a safe initial-arc and visible-progression policy for real new users (observing -> proving -> trusted as the agent earns it through success, approval acceptance, and eval scores), keeping `trusted` distinct from `ambient` and never auto-promoting past the safe ceiling without an explicit operator step. (2) The two-card Today queue on the Today surface, distinguishing "Waiting for your call" from "Executed and learned" (the executed-unattended audit already exists via `isSideEffectingTool` -> `is_unattended`), with a one-click undo on the executed card and the green-share metric trending as overrides fall. (3) The override-reason capture wired to write a memory signal through the existing `agent_memory` path, so an override compounds the moat rather than evaporating. See [`../features/trust-and-autonomy.md`](../features/trust-and-autonomy.md) for the arc UX contract.

**Acceptance criteria.** (1) A real new account starts at a defined initial arc and the UI shows the observing -> proving -> trusted progression with the agent's current position. (2) As an agent accrues clean runs, its arc advances on the documented signals and previously-gated `confirm` tools begin auto-running, with `review`-floor tools still gated. (3) The Today queue renders both card types; the executed card offers a working one-click undo. (4) An override prompts for a reason and that reason persists as a recallable `agent_memory` row. (5) The green executed-and-learned share is computed from real `agent_runs`/`agent_approvals` data, not a placeholder.

### B.6 · Dual-user MCP server and public API · M-D · status: Missing/Planned

**What.** Make "agent-friendly" true by shipping an MCP server and a documented public API so external agents and the user's other tools can integrate with Cadence's decision and memory layer. The typed A2A `HandoffPayload` and the agent card (`api/public/a2a.agents.cadence.card.ts`) exist internally; this exposes a real, documented surface. Defer build-your-own-agents and a marketplace; preserve the contract.

**Why.** Orchestration is commoditizing (MCP, A2A, Agent Skills) and the defensible asset is the proprietary outcome memory. Exposing it through MCP and a public API is both a differentiator and a fast-follower hedge (canon §14 names a B2B2B fallback: embed Cadence's memory layer inside Jira/Linear via MCP if the standalone window closes). Canon §8 and §12 M-D.

**Where in code.** (1) Add the `mcp_server` and `a2a` literals to the `CallSurface` union in `runtime.server.ts` so external-agent calls are metered and governed by the same chokepoint (the runtime contract already anticipates these surfaces; see [`../../architecture/runtime.md`](../../architecture/runtime.md)). (2) An MCP server endpoint under `src/routes/api/public/` that exposes a bounded, read-leaning tool set (decision read, memory query, mission status) over the existing server functions, authenticated per workspace and rate-limited. (3) A documented public API mirroring that surface for non-MCP callers, with the same auth and rate limits. (4) Every external call flows through `callModel`/`callModelStream` (no second path) so cost, guardrails, and `ai_events` telemetry apply uniformly. Reuse the `requireHookCaller`-style auth pattern but with per-workspace tokens, not the shared anon key.

**Acceptance criteria.** (1) An external MCP client lists and calls at least one Cadence tool and gets a governed, metered response (an `ai_events` row is written). (2) The public API serves the same surface with documented auth and a published rate limit. (3) An external agent reads a decision or queries memory through the public surface and the call is rate-limited and audited. (4) No external path bypasses the chokepoint; the new surfaces appear in the `CallSurface` union and in `ai_events.surface`. (5) The M-D exit holds: an external agent integrates successfully.

### B.7 · Pre-commit trace hook · M-B · status: Missing/Planned

**What.** A pre-commit hook (the build-time backstop for B.1) that scans staged text files and fails the commit on a banned dash (`U+2014`, `U+2013`) or any banned invisible Unicode character, so machine fingerprints never enter the repo. This is the authored-text counterpart to the runtime sanitizer: B.1 guards what the platform generates, this guards what we commit.

**Why.** The humanized-output rule is two-level (canon §13.8, [`../conventions/humanized-output.md`](../conventions/humanized-output.md)). The instruction layer drifts and a manual grep is forgettable; the durable backstop is a hook that blocks the commit. It keeps the convention enforced rather than aspirational.

**Where in code.** A Claude Code / git hook registered per [`../operations/hooks.md`](../operations/hooks.md), running the convention's two greps on staged text files (excluding code-fence and inline-code regions, and excluding generated files like `routeTree.gen.ts` and applied migration SQL): `rg "—|–"` must return zero, and the invisible-character `rg -nP "[\x{200B}\x{200C}\x{200D}\x{2060}\x{FEFF}\x{00A0}\x{202F}\x{00AD}\x{200E}\x{200F}\x{FFFD}]"` must return zero. The hook prints the offending file and line and the suggested replacement (period, comma, colon, parentheses, or a line break) and exits non-zero on any hit.

**Acceptance criteria.** (1) A staged file containing an em dash in prose blocks the commit with a clear, line-located message. (2) A staged file containing a zero-width space blocks the commit. (3) A legitimate dash inside a fenced code block or inline code does not block the commit. (4) Generated files and applied migration SQL are excluded so the hook never blocks on machine-authored artifacts. (5) The hook is documented in [`../operations/hooks.md`](../operations/hooks.md) and runs on every commit.
