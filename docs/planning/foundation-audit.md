## The one decision that gates step 1: tenancy

**0.1 is absent and everything in `plan.md` X2 (multi-product/workspace) + isolation tests depend on it.** Two paths:

- **(A) Retrofit now** ŌĆö add `workspace_id` + `product_id` to all 43 tables + rebuild all RLS policies to key on membership + the three keys, before building more features. Higher upfront cost; avoids a second migration of a larger schema later.
- **(B) Defer to step 5** ŌĆö keep single-key (`user_id`) through the first slice (DiscoverŌåÆDefineŌåÆPlan), retrofit when multi-product is built.

`plan.md` ┬¦3 sequencing rule ("architecture built right from step 1 so later stages are additions, not rewrites") argues for **(A)**. The cost is real but it compounds the longer it waits ŌĆö every new table and policy added under single-key is more to migrate. **Recommend (A), scoped as the first ticket.** _(Decision needed ŌĆö see open question Q-T1 below.)_
