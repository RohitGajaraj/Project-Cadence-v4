Circuit runs the full product lifecycle as one governed, autonomous loop. The scope below is the target product ŌĆö not a discovery tool. Each stage is run by autonomous agents; the human approves at the gates.

### Who we serve, what they expect, and the gap (grounding for the feature list)

The feature list below is a deliberate response to who the customer is and what the market already sets as table-stakes ŌĆö not a wish list. Inspiration drawn from products worth learning from is captured in [`docs/references/competitive-reference.md`](docs/references/competitive-reference.md) (deferred; inspiration only).

**Personas (who orchestrates Circuit):**

- **P1 ŌĆö Enterprise Director / VP of Product (The Portfolio Governor)**: owns portfolio-level alignment, agent spend budgets, and compliance. Pain: Swarm drift, lack of visibility into agent actions, security audits, budget overruns, and multi-product roadmap health.
- **P2 ŌĆö Lead / Senior PM (The Daily Cockpit Operator)**: runs day-to-day discovery, spec definition, and feature delivery. Pain: Mechanical process work (spec writing, ticket updates, release logs, alert triage) eating up time needed for strategic judgment.
- **P3 ŌĆö Engineering Lead / Tech Architect (The Code Gatekeeper)**: guards repository health, code quality, and branch stability. Pain: Agent-written code drifting from internal architecture guidelines, introducing regressions, or breaking sandboxed environments.
- **P4 ŌĆö UX/UI Designer (The Visual Validator)**: ensures design tokens and brand style guide alignment. Pain: Generated code failing to match design system tokens or looking off-brand in mockups.
- **P5 ŌĆö GTM Lead / Product Marketer**: handles positioning and product launch. Pain: Generating changelogs, marketing collateral, newsletter posts, and distribution updates by hand for every release.
- **P6 ŌĆö Customer Success & Support Lead**: monitors product support feedback. Pain: High ticket volumes, slow bug triage, and user signals getting lost before reaching the product manager.

**Market expectations (table-stakes ŌĆö must-haves, or we're not credible):** auth + SSO-ready tenancy; real (not mocked) data; connectors to the tools they live in; an assistant/chat surface; observability + cost visibility; approvals/audit for anything autonomous; multi-project support; export/no-lock-in.

**The gap (where the market is underserved ŌĆö our opening):** point tools own single bands (engineering autonomy, or issue tracking, or research synthesis). **No one runs the _whole_ lifecycle ŌĆö discover ŌåÆ build ŌåÆ ship ŌåÆ launch ŌåÆ support ŌåÆ learn ŌĆö as one governed, autonomous, multi-product loop.** That gap is the spine of the feature catalog in section 2.

**What the customer actually wants (the job):** "Take this intent and run it end-to-end ŌĆö autonomously, in parallel, across my products ŌĆö and only stop me for the calls that matter." Every feature below maps to that job.

**The feature list derives from the PM's actual day.** The product manager is the cockpit operator ŌĆö the features below exist to run the PM's day (triage inbox ŌåÆ decide ŌåÆ synthesize ŌåÆ spec ŌåÆ tickets ŌåÆ build/test/ship ŌåÆ launch/update ŌåÆ answer "why is this on the roadmap?" ŌåÆ report ŌåÆ repeat). The day-to-features mapping and the PM/eng-lead lenses are in [`docs/planning/considerations.md`](docs/planning/considerations.md). If a feature does not serve a real PM task, question it.

### Lifecycle stages (the 12-Stage Product Loop)

**S1 ŌĆö Signal Capture**

- Multi-source signal ingest: support tickets, churn, usage telemetry, sales notes, reviews, NPS, interviews.
- Agent clustering ŌåÆ themes ŌåÆ ICE-scored opportunities; continuous (not one-shot) synthesis; dedupe by similarity.
- A continuously-updated discovery feed per product.

**S2 ŌĆö Audio Sync (WhisperFlow)**

- High-context audio recording, transcription, and meeting spec extraction.
- Translates voice memos or team meetings into feature specs and backlog tickets with zero friction.

**S3 ŌĆö ICE Prioritization**

- Prioritization queue management: opportunity scoring based on Impact, Confidence, and Ease.
- Automated re-prioritization as new signals arrive or existing ones scale in severity.

**S4 ŌĆö Spec Definition**

- Opportunity ŌåÆ PRD/spec, retrieval-grounded and cited; inline `/ai` editing; versioned with diff.
- Acceptance criteria, non-goals, risks, success metrics auto-drafted; human edits and approves.

**S5 ŌĆö Sprint Planning**

- Spec ŌåÆ dependency-aware task graph with estimates, owners, risk flags.
- Outcome-oriented roadmap (Now/Next/Later) synced automatically to Linear/Jira.

**S6 ŌĆö Agentic Build**

- In-product multi-file coding: Builder Agent claims file locks, checks out branches, and writes code.
- Surgical modifications across files, committing atomically and resolving conflict flags.

**S7 ŌĆö Visual QA**

- Sandbox preview environments generated automatically for visual validation of UI changes.
- Let designers and PMs audit layout and styling against design system tokens before merging.

**S8 ŌĆö Safe Release**

- Merges PRs, runs deploy checklists, deploys, and monitors production health.
- Automated rollback triggered if error metrics or exception counts spike post-deploy.

**S9 ŌĆö GTM Launch**

- Growth agent drafts release changelogs, marketing blog posts, newsletter copies, and announcement text.

**S10 ŌĆö Support Triage**

- Support agent triages incoming customer tickets, drafts answers, and routes bug logs back into Signal Capture (S1).

**S11 ŌĆö Cohort Analytics**

- Automated feature adoption tracking, cohort analysis, and outcome funnel monitoring.

**S12 ŌĆö Learn & Reflect**

- Outcome evaluation: compares performance telemetry with the original PRD spec.
- Updates the Product Memory Graph and publishes a learning brief to the portfolio dashboard.

### Cross-cutting platform (mandatory, spans all stages)

**X1 ŌĆö Orchestration & automation engine**

- Workflow definitions across stages; event/schedule/webhook triggers; an automation builder.
- **Many sub-agents running in parallel** within a mission; **many agent sessions running in parallel** across missions.
- A live orchestration view (the mission/agent graph) with per-node status, cost, and approval state.
- Planner/executor loop with max-step/max-cost caps, replay-and-branch, cancellation, checkpoints.

**X2 ŌĆö Multi-product & multi-workspace**

- **Multiple products in flight ŌĆö Product A/B/C ŌĆö grouped under Workspaces (A/B/C).**
- Per-product/workspace isolation of data, agents, memory, budgets, and access (RLS by `user_id` + `workspace_id`; see [`architecture/data.md`](./architecture/data.md)).
- Product switcher; cross-product portfolio view.

**X3 ŌĆö Trust & governance (the AI trust stack)**

- Per-call telemetry, LLM-as-judge scoring, guardrails (input + output), RAG with citations, prompt versioning + A/B, eval harness with regression gate, drift watch, budgets, live trace waterfall, incidents log, approval gates everywhere.

**X4 ŌĆö Product Memory**

- Queryable graph of signals ŌåÆ themes ŌåÆ opportunities ŌåÆ decisions ŌåÆ experiments ŌåÆ outcomes, with `supersedes` lineage; the institutional context agents reason over; exportable as skill packs.

**X5 ŌĆö Interop (agent-native)**

- MCP server (Circuit as a tool surface) + MCP client (consume external tools) + A2A (Agent Cards, delegate-to-agent, peer registry). Capability scopes, rate limits, prompt-injection guard on external results, audit log.

**X6 ŌĆö Identity, auth, billing**

- Auth (email + OAuth), session management, per-workspace membership (future: roles/teams), BYO model keys (encrypted), usage/billing metering against budgets. See [`architecture/security.md`](./architecture/security.md).

## 2. Granular feature catalog (login to agent-transfer, with what + why)

> **Inspiration, not comparison.** This catalog absorbs the best ideas from products worth learning from ŌĆö factory.ai's autonomous build/ship + live "mission" view (we extend it past code into launch/support), hyperagent's team-of-agents with per-agent budgets and self-improving memory, and Linear's agents-as-first-class-users + saved-workflow skills + event automations. ### A. Identity & access

- **Sign up / onboarding** ŌĆö email+password + Google OAuth; create first workspace + product; set display name, timezone, working hours. ŌĆö _Entry point; personalization and scheduling depend on it._ `[reuse]`
- **Login / logout** ŌĆö authenticated session, sign-out everywhere, session refresh, `Last-Event-ID` resume for streams. ŌĆö _Baseline security and reliable streaming._ `[reuse]`
- **Password reset / email verification** ŌĆö standard flows; verification on by default. ŌĆö _Account safety._ `[reuse]`
- **Profile & preferences** ŌĆö display name, role, timezone, working hours, default model per surface. ŌĆö _Drives greeting, scheduling, model routing._ `[reuse]`
- **BYO model keys** ŌĆö add/test/rotate/delete provider keys, encrypted; masked display. ŌĆö _Model-agnostic; cost control; enterprise trust._ `[reuse]`
- **Roles & membership (future)** ŌĆö invite, per-workspace roles, SSO/SAML. ŌĆö _Team and enterprise readiness._ `[new]`

### B. Workspaces & products

- **Create / switch workspace** ŌĆö top-level tenancy boundary. ŌĆö _Isolates a client/company/initiative._ `[extend]`
- **Create product under workspace** ŌĆö Product A/B/C inside a workspace; vision/problem/target-users/metrics/stage. ŌĆö _Run several products in parallel without bleed._ `[extend]`
- **Product switcher + portfolio view** ŌĆö fast switch; cross-product overview. ŌĆö _One operator, many products._ `[new]`

- **Archive / delete product** ŌĆö lifecycle hygiene. ŌĆö _Avoid clutter; data governance._ `[new]`

### C. Agents ŌĆö configuration

- **Agent roster** ŌĆö list durable agents + spawned sub-agents per product. ŌĆö _See the team you command._ `[extend]`
- **Create / clone / configure agent** ŌĆö system prompt, tool allow-list, model, temperature/top*p/seed, max tokens, schedule, approval mode, memory toggle. ŌĆö \_Tailor behavior + governance per role.* `[reuse]`

### E. Agent communication, coordination & transfer (mandatory)

## 6. Product-agent roster (re-examined: explicit vs. implicit)

**The question:** is a fixed roster of 16 named agents right? **Answer:** no ŌĆö split it. Use a small set of **durable, named lifecycle agents** (explicit, one owner per stage) plus an orchestrator that **dynamically spawns ephemeral specialist sub-agents** for sub-tasks (implicit). Fixed seats for micro-roles create clutter; dynamic sub-agents scale with the work.

**6 outcome-focused durable agents (explicit, one per lifecycle band):**

The roster is kept narrow and delivery-oriented. Each agent owns a clear band of the product lifecycle. Fixed seats for micro-roles are avoided ŌĆö those collapse into ephemeral specialists.

| Agent            | Lifecycle band                                                  | Owns                                                                                                             | Primary pain point addressed                                    |
| ---------------- | --------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| **Orchestrator** | All stages                                                      | Mission coordination, spawns ephemeral specialists, manages approval routing, parallelism                        | Coordination overhead, context loss between stages              |
| **Discovery**    | S1 Signal Capture + S2 Audio Sync                               | Signal Ingest (Zendesk/Intercom/Slack), clustering opportunities, WhisperFlow transcription and audio extraction | Shallow discovery, anecdotal signals, missing verbal alignments |
| **Strategist**   | S3 ICE Prioritization + S4 Spec Definition + S5 Sprint Planning | Opportunity scoring, cited PRD generation, Linear/GitHub Issues sync, dependency-aware task graphs               | Spec drafting overhead, weak prioritization, feature factories  |
| **Builder**      | S6 Agentic Build + S7 Visual QA + S8 Safe Release               | Multi-file coding, file claims ledger, sandbox previews, PR merge checks and deploy monitoring                   | Developer bottlenecks, styling drift, release quality checks    |
| **Growth**       | S9 GTM Launch                                                   | Outbound changelogs, blog posts, Slack notifications and release announcements                                   | Launch overhead, delayed go-to-market communication             |
| **Analyst**      | S10 Support Triage + S11 Cohort Analytics + S12 Learn & Reflect | Support ticket routing to S1, adoption funnels, performance metrics vs PRD specs, Product Memory                 | Closed-loop support, un-measured features, memory decay         |

**Ephemeral specialists (spawned by Orchestrator per task):** competitor scan, UX critique, stakeholder brief, data analysis, security review, code review, and any other short-lived specialist. They inherit the chokepoint, tool allow-list, and governance. No permanent seats for micro-roles.

**Persona mapping:**

- **P1 (Director / VP of Product)**: strategic oversight, budget controls, audit log reviews, portfolio health, uses Analyst + Orchestrator.
- **P2 (Lead / Senior PM)**: daily operator, runs swarm briefs, PRD editor, discovery feeds, uses Discovery + Strategist + Analyst.
- **P3 (Engineering Lead)**: code review, Builder agent claims review, CI verification, uses Builder.
- **P4 (UX/UI Designer)**: sandbox preview validation, style token audit, uses Builder (Visual QA).
- **P5 (GTM Lead)**: launch copywriting, distribution approvals, uses Growth.
- **P6 (Support Lead)**: triage validation, bug signal loops, uses Analyst.

Schema per agent: `slug, name, system_prompt, tool_allowlist[], default_model, temperature/top_p/seed, max_tokens, schedule_cron?, approval_mode (auto|confirm|review), memory_enabled`. Roster changes need a migration + a build-order entry.

## 8. Lessons learned (kept from the legacy build)

One AI chokepoint paid for itself (judge added in half a day). Loader + `useSuspenseQuery` killed `useEffect+fetch` regressions. OKLCH tokens ended dark-mode contrast bugs. Bugs fixed: name-vs-email leak; duplicate agent fan-out (advisory lock); action-item due-date glitch; Notion BFS reorder; trace recursion blow-up; cache cross-user poisoning (salted key); CSV params cap (chunked inserts). Known limits: clustering degrades past ~1000 signals/run; Studio truncates files >2k lines; Docs sync drops comments; `model_pricing` hand-maintained.
